**Bug 1 — Reset time parsing:**

1. Add a `rateLimitResetMessage: String?` property to `TokenRecord` (SwiftData @Model). This stores the raw message like `"You're out of extra usage · resets 4am (Europe/Berlin)"` so UsageWindowTracker can parse the actual reset time.

2. Update `SessionWatcher.swift` — when creating TokenRecord instances (in both `pollForChanges` and `processSessionFile`), pass through `record.rateLimitResetMessage` to the new TokenRecord property.

3. In `UsageWindowTracker.swift`, replace the hardcoded `limitTime.addingTimeInterval(windowDuration)` reset time calculation with a new method `parseResetTime(from message: String, relativeTo date: Date) -> Date?` that:
   - Uses regex to extract time and timezone from the message: pattern like `resets\s+(\d{1,2}(?::\d{2})?\s*(?:am|pm)?)\s*\(([^)]+)\)` (case-insensitive)
   - Examples to handle: "resets 4am (Europe/Berlin)", "resets 4:00am (Europe/Berlin)", "resets 16:00 (UTC)"
   - Creates a `DateComponents` with the parsed hour (and minute if present), uses `TimeZone(identifier:)` for the timezone string
   - Computes the next occurrence of that time — if parsed time is in the past today, it means tomorrow
   - Falls back to `limitTime + 5h` if parsing fails

4. In the `evaluate()` method, find the reset message: query `lastRateLimitMessage` from the Session that has the most recent rate limit, or from TokenRecord. Use `parseResetTime(from:relativeTo:)` to get the actual resetTime. Replace `let resetTime = limitTime.addingTimeInterval(windowDuration)` with `let resetTime = parseResetTime(from: message, relativeTo: limitTime) ?? limitTime.addingTimeInterval(windowDuration)`.

5. The `windowDuration` constant should remain as fallback but no longer be the primary reset time source.

**Bug 2 — Project discovery:**

6. In `SessionWatcher.swift`, fix `deriveProjectPath(from:)`. The current approach `dirName.replacingOccurrences(of: "-", with: "/")` is wrong because project names like `ClaudeCode-skills-installieren` contain dashes that are NOT path separators.

   The Claude projects directory encoding uses `-` as separator for `/`. The directory name `-Users-felixleis-ClaudeCode` represents `/Users/felixleis/ClaudeCode`. The problem: there's no way to distinguish `-` as path separator from `-` in a directory/project name using simple replacement.

   Fix approach: Instead of trying to decode the directory name, validate against the filesystem. Replace `deriveProjectPath` with logic that:
   - Takes the directory name (e.g., `-Users-felixleis-ClaudeCode-skills-installieren`)
   - Removes the leading `-` to get `Users-felixleis-ClaudeCode-skills-installieren`
   - Splits by `-` to get components: `["Users", "felixleis", "ClaudeCode", "skills", "installieren"]`
   - Tries progressively longer path combinations from the left, checking which ones exist on disk using `FileManager.default.fileExists(atPath:)`:
     - `/Users` (exists, continue)
     - `/Users/felixleis` (exists, continue)
     - `/Users/felixleis/ClaudeCode` (exists, continue)
     - `/Users/felixleis/ClaudeCode/skills` — check if exists. If NOT, try joining with dash: `/Users/felixleis/ClaudeCode-skills` — check that. Continue with the strategy that works.
     - The key insight: at each step, try `nextComponent` as a new path segment first. If that doesn't exist as a directory, try appending `-nextComponent` to the current last segment.
   - This greedy approach correctly resolves both `-Users-felixleis-ClaudeCode` -> `/Users/felixleis/ClaudeCode` AND `-Users-felixleis-ClaudeCode-skills-installieren` -> the actual path on disk.
   - Cache results in a `[String: String]` dictionary on SessionWatcher to avoid repeated filesystem checks.
